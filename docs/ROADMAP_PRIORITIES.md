# Priority Roadmap

> 目的：將 AI-First Framework 從「設計完整的框架」推進成「低摩擦、可驗證、可量化、可被團隊穩定採用的產品」。
>
> 本文件聚焦 5 個優先補強方向，定義每個方向的問題、目標、近期交付物與完成判準。

---

## P0 — 定義最小可用版本（Lite Mode）

### 為什麼先做

目前框架流程完整，但對 1-2 人團隊或首次導入者來說，進入成本偏高。若沒有清楚的最小可用路徑，團隊很容易因為文件與流程密度而放棄採用。

### 目標

定義一條能在小團隊中真正跑起來的最短主路徑，保留框架核心價值，但明確刪去早期非必要負擔。

### 近期交付物

1. `docs/LITE_MODE.md`
2. Lite Mode 啟動條件與限制說明
3. 精簡版必備產出清單
4. Lite Mode 對應的 slash command 或啟動話術
5. 新專案與舊專案各一條 Lite 入口

### 最小範圍建議

- 保留：Task-Master、STATE、TASKS、Gate、標準文件 SSOT
- 精簡：非關鍵文檔、非必要 Agent 切換、過早的完整合規產出
- 原則：先能持續交付，再逐步升級回完整模式

### 完成判準

- 新使用者 15 分鐘內可理解如何開始
- 1 人可完成第一個 feature 的最小閉環
- README 與 CLAUDE.md 都有明確 Lite 入口

## P1 — 把核心規則轉成可驗證腳本

### 為什麼第二優先

目前很多規則存在於文件、SEED 與 SKILL 中，仍偏向約定式執行。若缺少機器驗證，流程容易在實際使用時逐步漂移。

### 目標

將最關鍵、最常違反、最容易自動檢查的規則收斂成 workflow checks，讓框架從 SOP 升級成半強制系統。

### 近期交付物

1. `tools/workflow-test/` 新增或擴充規則檢查
2. `scripts/validate-framework.sh` 或等價入口
3. CI 中加入核心結構驗證
4. 規則分級：error / warning / info

### 第一批應機器化的規則

- 必要文件是否存在
- `TASKS.md` / `STATE.md` 關鍵欄位是否完整
- 命名是否符合 F-code / 文件規則
- `10_Standards/` 是否存在且路徑正確
- `context-seeds/` 與 `context-skills/` 核心檔是否齊全

### 完成判準

- 使用者能一個指令完成框架健康檢查
- 主要結構錯誤能在進入 Pipeline 前被發現
- CI 可以阻擋明顯破壞框架契約的修改

## P2 — 收斂核心文件邊界，降低同步成本

### 為什麼第三優先

`TASKS.md`、`MASTER_INDEX.md`、`memory/STATE.md`、`memory/decisions.md` 都有明確價值，但長期維護時容易資訊重疊、更新責任不清、欄位漂移。

### 目標

定義每份核心文件的唯一責任邊界，避免同一件事在不同地方重複維護。

### 近期交付物

1. `docs/INFORMATION_ARCHITECTURE.md`
2. 核心文件責任矩陣
3. 「什麼資訊寫在哪裡」範例表
4. Agent 更新責任說明

### 建議邊界

- `memory/STATE.md`：只記錄當前工作態與恢復點
- `TASKS.md`：只記錄任務、狀態、交接摘要
- `MASTER_INDEX.md`：只記錄正式產出文件與成熟度
- `memory/decisions.md`：只記錄架構與流程決策

### 完成判準

- 新增一個 feature 時，團隊知道該更新哪幾份檔案
- 同一資訊不需要在 3 個以上地方重複維護
- Agent handoff 規則更清楚，不靠個人理解補齊

---

## P3 — 建立端到端示範專案

### 為什麼第四優先

目前框架文件說明已經相對完整，但示範型證據仍不足。沒有端到端案例，使用者很難快速理解「這套框架實際跑完一次會長什麼樣」。

### 目標

建立一個足夠小、但能覆蓋核心流程的 reference project，讓新使用者可以直接照著走。

### 近期交付物

1. `examples/` 或 `demo-project/`
2. 一個從 F01 到 Gate 3 的完整示例
3. 對應的產出文件樣本
4. README 中的 walkthrough

### 示例建議

- 題目：登入 / 任務管理 / 簡單 CRUD 類功能
- 範圍：至少涵蓋 P01、P02、P03+P04、Gate 1/2/3
- 要求：能看見文件、程式碼、測試、交接摘要彼此如何連動

### 完成判準

- 新使用者可用示範專案反推框架用法
- 團隊成員可拿示例當模板複製
- 新增功能時不需只靠抽象規範理解流程

---

## P4 — 強化採用體驗與入口設計

### 為什麼要做

框架概念完整，但入口很多，初次使用者需要先理解大量術語與流程，才能知道下一步。這會降低 adoption。

### 目標

把常用入口收斂成少數高頻路徑，讓使用者在不知道全貌時也能先開始。

### 近期交付物

1. README 首屏入口重構
2. `CLAUDE.md` 首次使用區塊再簡化
3. 「我現在該做什麼」導覽頁
4. command / pipeline / agent 的選擇指南

### 優先優化的 5 個入口

1. 建新專案
2. 接入舊專案
3. 開始下一件事
4. 交接
5. Hotfix

### 完成判準

- 使用者不需要先讀完整框架就能開始第一步
- 最常見操作都能在 1-2 個入口內找到
- README 不再主要是概念陳述，也能直接導向行動

---

## 建議執行順序

1. P0 Lite Mode
2. P1 規則腳本化
3. P2 文件邊界收斂
4. P3 端到端示範專案
5. P4 採用體驗優化

> 原則：先降低進入成本，再提高約束力，再補齊教學與示例，最後優化入口體驗。

---

## 建議里程碑

### Milestone A — 可開始

- Lite Mode 定義完成
- README 有 Lite 入口
- 健康檢查可驗證核心結構

### Milestone B — 可穩定使用

- 文件責任邊界明確
- 常見錯誤可由測試工具發現
- 使用者不需反覆猜測應更新哪些檔案

### Milestone C — 可推廣

- 端到端示例完整
- Adoption 入口清楚
- 新團隊可在低輔導成本下導入

---

## 成功指標建議

- 新專案初始化到第一個 feature 開始的平均時間
- 使用者完成第一次 pipeline 的成功率
- `workflow-test` 發現的結構性錯誤數
- Lite Mode 與完整模式的使用比例
- 首次使用者在 README 停留後的啟動成功率
