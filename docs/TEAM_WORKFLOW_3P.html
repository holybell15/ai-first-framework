為什麼垂直切？

一個人 + Claude 就能跑完整條 Pipeline（Claude 扮演所有 12 個 Agent）。三人的價值不是分 Pipeline 階段，而是三條 Feature 同時推進。

兩年節奏

1

Month 1-2：共治層建立

三人一起做前 2-3 個核心 Feature。建立 ADR、DB Schema 基礎、Design System。這段時間是水平切，因為要對齊標準。

2

Month 3-24：各自認領 Feature

共治層穩定後，切換為垂直切。每人認領 Feature，並行推進。每兩週 Sprint Planning 對齊優先級。Gate Review 交叉做。

重要：前兩個月不要急著分開跑。共治層（架構、DB、設計系統）沒對齊就分開，後面三個人的產出會不一致，整合時代價更大。

Teams Webhook 設定（5 分鐘）

只需要做一次，之後所有交接自動通知。

1

在 Teams 建立 Incoming Webhook

Teams 頻道 → ... → Connectors（或 Workflows）→ 搜尋 Incoming Webhook → Configure → 命名為 AI-First Handoff → 複製 Webhook URL

Teams 新版（Workflows）：如果看不到 Connectors，改用 Workflows → Post to a channel when a webhook request is received → 選頻道 → 複製 URL
2

貼到 .env

# ── Teams 交接通知（主要管道）──────────────── HANDOFF_TEAMS_WEBHOOK=https://xxx.webhook.office.com/webhookb2/...

放在專案根目錄的 .env 檔案裡。不要 commit .env。

3

設定 GitHub Actions 備援

GitHub repo → Settings → Secrets → New secret → Name: HANDOFF_TEAMS_WEBHOOK → 貼上同一個 URL

!

測試

./scripts/handoff.sh Test Test F00 "Webhook 測試"

如果 Teams 頻道收到通知 → 設定完成。

協作鐵規

違反任何一條都會造成混亂或品質問題。

🔒

Gate Review 必須開新 Session

寫 code / 寫規格的 session 不做 Review。新 session = 新鮮眼睛，避免確認偏誤。

🔀

Gate Review 交叉執行

寫的人不審自己。Alice 寫的 → Bob 或 Carol 審。三人交叉確保獨立性。

📋

TASKS.md 只 Append

不修改他人已寫的行。衝突以時間較晚者為準。這是防止覆蓋他人進度的保險。

📝

STATE.md 單人寫入

只有 active_member 有寫入權。其他人唯讀。避免狀態互相覆蓋。

🔗

每次交接必 Commit + 通知

沒有 commit 就沒有交接。下一個人 git pull 看不到你的產出 = 等同沒做。

🚫

decisions.md 不直接 Push

技術決策透過 PR 修改。三人都要看到變更，不能一個人偷偷改架構決策。

⚡

G4-ENG 必須通過才能 P04

沒有通過工程設計驗收，不允許開始寫 code。先驗證設計，再動手實作。

🌊

Wave 並行寫入前先讀最新版

兩人並行時（如 Architect ∥ DBA），寫入共享文件前先讀取最新版，防止覆蓋。

📐

修改程式碼前必做 Grounding

執行 /ground 讓 code-grounder agent 先理解現況。禁止假設、禁止猜測。
