## 📊 F-code 分配登記

> 每次啟用新 Feature，先在此登記 F-code，防止重複分配。

| F-code | 功能名稱 | 中文說明 | Lifecycle | Feature Owner | Decision Owner | 啟動日期 |
|--------|---------|---------|-----------|---------------|----------------|---------|
| F00 | [系統層級 / 共用] | 系統共用功能 | — | — | — | — |

> 格式範例：`| F02 | IncomingCall | 來電彈屏 | 開發中 | 王小明(BE) | 李小華(PM) | 2026-03-10 |`

### Feature Lifecycle 狀態定義

| 狀態 | 對應 Pipeline | 進入條件 | 說明 |
|------|-------------|---------|------|
| `需求定義中` | P01 進行中 | F-code 登記後 | Interviewer/PM/UX 執行中 |
| `需求已收斂` | Gate 1 通過 | Gate 1 Pass | 需求完整，可進入技術設計 |
| `設計中` | P02-P03 進行中 | Gate 1 Pass | 架構/API/DB 設計中 |
| `工程就緒` | G4-ENG 通過 | G4-ENG Pass | 規格完整，可直接實作 |
| `開發中` | P04 進行中 | G4-ENG Pass | 功能實作 + 測試中 |
| `驗證中` | Gate 3 + P05 | P04 完成 | 品質驗證 + 合規審查中 |
| `已發佈` | P06 完成 | 上線確認 | 功能已上線運行 |
| `blocked` | 任何階段 | 遇到阻塞 | ⚠️ 附加標記，需說明阻塞原因和解除條件 |

> `blocked` 不是獨立狀態，而是附加在任何 lifecycle 狀態上的標記。例如：`設計中 (blocked)` 表示設計階段被阻塞。

**狀態更新時機**：每次 Gate 通過或 Pipeline 階段切換時，更新對應 F-code 的 Lifecycle 欄位。
**Blocked 標記**：遇到阻塞時加上 `(blocked: 原因)`，解除後移除標記。

### Feature Owner 制度

| 角色 | 職責 | 何時指定 |
|------|------|---------|
| **Feature Owner** | 功能的工程執行負責人，確保從 P02 到 P06 的技術產出品質 | F-code 登記時指定 |
| **Decision Owner** | 功能的需求與範圍決策者，負責 Confirm 點和 Gate 真人審核 | F-code 登記時指定 |

- Feature Owner 通常是主要實作工程師（前端或後端）
- Decision Owner 通常是 PM 或系統分析
- 一個人可以同時擔任多個功能的 Owner
- 1-2 人模式下，同一人兼任兩個角色

## CI 維護規則

- **新增文件**：Agent 產出後立即在對應分類加一行
- **版本升級**：修改「版本」欄，不新增行
- **Gate 通過**：更新「成熟度」欄為 `Baselined`，「最後 Gate」欄填入 Gate 代號；同步更新 F-code 的 Lifecycle 狀態
- **文件棄用**：成熟度改為 `Deprecated`，不刪行（保留追蹤記錄）
- **F-code 分配**：新功能討論確認後先登記 F-code + 指定 Feature Owner + Decision Owner，再啟動 Pipeline
- **Lifecycle 更新**：Pipeline 階段切換或 Gate 通過時，更新 F-code 表的 Lifecycle 欄位
