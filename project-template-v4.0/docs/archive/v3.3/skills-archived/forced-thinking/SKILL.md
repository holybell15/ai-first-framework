# 強制思考模板（Forced Thinking Templates）

## 為什麼是「強制」？

- **AI 的盲點**：AI 擅長生成，但不擅長「停下來想」。強制提問讓每個產出都經過關鍵思考
- **人類的盲點**：趕時程時容易跳過風險評估。結構化清單確保不遺漏
- **下游成本**：需求階段漏想的問題，到實作階段修復成本是 10 倍以上
- **治理要求**：每個強制思考的產出都是 Gate Review 的審查依據

---

## 模板 1：需求思考（P01 — 每個功能必答）

> **觸發時機**：PM Agent 撰寫 User Story / AC 前，對每個功能逐一回答。
> **產出位置**：嵌入功能需求文件（`02_Specifications/US_F##_*.md`）頂部。
> **Gate 1 審查**：Review Agent 確認每個功能都有完成此模板。

### 必答 7 問

```markdown
## 需求思考紀錄 — [功能名稱]

### RT-01 使用者為什麼需要這個功能？
> [回答：描述使用者的痛點或目標，不是系統的功能描述]

### RT-02 這個功能解決什麼問題？
> [回答：具體問題，不是「提升體驗」這類空話]

### RT-03 如果這個功能失敗或異常，會造成什麼影響？
> [回答：對使用者、對業務、對資料的影響]

### RT-04 不同角色使用這個功能時有差異嗎？
> [回答：列出角色差異，或明確標記「無差異」]

### RT-05 這個功能有權限限制嗎？
> [回答：誰可以操作、誰不行、未授權時顯示什麼]

### RT-06 這個功能涉及狀態轉換嗎？
> [回答：列出狀態流轉，或明確標記「無狀態」]

### RT-07 最壞情境是什麼？
> [回答：最極端的使用方式、最大的資料量、最糟的網路環境]
```

### 判定規則

| 狀態 | 條件 |
|------|------|
| ✅ 通過 | 7 問全部有實質回答（不是「無」或「N/A」敷衍） |
| 🟡 待補 | 1-2 問回答不充分，標記待 Gate 1 前補齊 |
| 🔴 阻塞 | 3 問以上無法回答 → 退回 Interviewer 補充訪談 |

## 模板 1.5：Plan Challenge（P01→P02 銜接 — Gate 1 通過後必答）

> **觸發時機**：Gate 1 通過後、P02 技術設計開始前。
> **產出位置**：嵌入 `03_System_Design/F##-SW-ARCH.md` 頂部。
> **目的**：在投入設計前先挑戰「該不該做」和「做多大」。
> 靈感來源：gstack /plan-ceo-review — 在寫 code 前挑戰 scope 和方向。

### 必答 5 問

```markdown
## Plan Challenge — [功能名稱]

### PC-S01 這個功能是最小可行的解法嗎？
> [回答：有沒有更簡單的方式達到同樣目的？能否拆分？]
> 🔴 如果回答「不確定」→ 退回 PM 釐清最小 scope

### PC-S02 Blast Radius 評估：預計修改幾個檔案？
> [回答：預估檔案數。> 8 個檔案 = 🟡 smell，考慮拆分]
> 判定：1-4 檔案 ✅ / 5-8 🟡 / 9+ 🔴 建議拆 Feature

### PC-S03 有沒有現成的東西可以重用？
> [回答：Search Before Building — 框架內 Skill、npm 套件、既有模組]
> 必須列出至少搜尋過的 3 個地方（context-skills/、10_Standards/、npm）

### PC-S04 這個設計方向，6 個月後還適用嗎？
> [回答：是否為 one-off hack 還是可持續的架構？]
> 不需要 over-engineer，但需要確認不會在下個 Sprint 就要重寫

### PC-S05 如果今天不做，最壞會怎樣？
> [回答：延遲的風險和成本。如果答案是「沒事」→ 重新評估優先級]
```

### 判定規則

| 狀態 | 條件 |
|------|------|
| ✅ 通過 | 5 問都有實質回答，Blast Radius ≤ 8 檔案 |
| 🟡 待調 | PC-S02 > 8 檔案，或 PC-S03 未充分搜尋 → 調整 scope 後繼續 |
| 🔴 阻塞 | PC-S01 答「不確定」或 PC-S05 答「沒影響」→ 退回 PM 重新評估 |

## 模板 2：設計思考（P02/P03 — 每個功能必答）

> **觸發時機**：Architect / Backend / Frontend Agent 設計功能前。
> **產出位置**：嵌入設計文件（`03_System_Design/F##-SW-ARCH.md` 或 `F##-API.md`）。
> **G4-ENG 審查**：Review Agent 確認每個功能都有完成此模板。

### 必答 8 問

```markdown
## 設計思考紀錄 — [功能名稱]

### DT-01 是否影響既有資料？
> [回答：會修改哪些表/欄位，或明確標記「不影響」]

### DT-02 是否影響既有流程？
> [回答：哪些現有功能的行為會改變]

### DT-03 是否影響外部系統？
> [回答：第三方 API、檔案系統、訊息佇列等]

### DT-04 是否需要交易控制（Transaction）？
> [回答：哪些操作必須原子性完成，失敗時如何處理]

### DT-05 是否需要重試機制？
> [回答：哪些外部呼叫可能失敗，重試策略是什麼]

### DT-06 例外處理策略是什麼？
> [回答：每種錯誤的處理方式 — 重試/降級/通知/中斷]

### DT-07 有效能風險嗎？
> [回答：大量資料/高併發/慢查詢/大檔案上傳等]

### DT-08 需要回滾策略嗎？
> [回答：部署失敗/資料錯誤時如何復原]
```

### 判定規則

| 狀態 | 條件 |
|------|------|
| ✅ 通過 | 8 問全部有實質回答 |
| 🟡 待補 | 1-2 問標記「待確認」，但有確認計畫和時程 |
| 🔴 阻塞 | DT-01/02/03 任一無法回答 → 退回需求釐清 |

## 模板 3：提交前檢查（P04 — 每次 commit 前必答）

> **觸發時機**：Backend / Frontend Agent 每次 commit 前。
> **產出位置**：commit message 或 `08_Test_Reports/F##-commit-checks.md`。
> **Gate 3 審查**：Review Agent 抽查 commit 是否有對應的提交前檢查。

### 必答 6 問

```markdown
## 提交前檢查 — [功能/修改描述]

### PC-01 是否處理了例外情境？
> [回答：列出已處理的例外，或說明為何無需處理]

### PC-02 是否有記錄必要的 log？
> [回答：哪些操作有 log、log 層級、是否含 PII]

### PC-03 邊界條件是否已檢查？
> [回答：空值/null、超長字串、極端數量、併發等]

### PC-04 是否可能影響既有功能？
> [回答：列出可能受影響的功能，或說明為何無影響]

### PC-05 是否已對照需求追溯？
> [回答：此修改對應哪個 AC / User Story]

### PC-06 是否補齊測試？
> [回答：新增/修改了哪些測試，或說明為何無需新測試]
```

### 執行方式

- **Quick Mode（§37）修改**：PC-01 + PC-04 必答，其餘可簡答
- **正常修改**：6 問全答
- **跨模組修改**：6 問全答 + 額外標記受影響模組清單

### 判定規則

| 狀態 | 條件 |
|------|------|
| ✅ 可提交 | 必答項全部有實質回答 |
| 🟡 補充後提交 | 有遺漏但非高風險 |
| 🔴 不可提交 | PC-04 回答「有影響」但未列出影響範圍 |

## 模板 4：缺陷學習（QA 發現缺陷 / Hotfix 事後檢討時必填）

> **觸發時機**：QA Agent 記錄缺陷時、Hotfix Step 7 事後檢討時。
> **產出位置**：`08_Test_Reports/F##-TR.md` 的缺陷區塊，或 `memory/hotfix_log.md`。
> **回顧改善**：retro skill 彙整所有缺陷學習紀錄，形成改善建議。

### 必答 4 問

```markdown
## 缺陷學習紀錄 — [缺陷描述]

### DL-01 問題根因是什麼？
> [回答：不是「程式寫錯」，而是為什麼會寫錯 — 需求不清？設計遺漏？測試不足？]

### DL-02 哪一個階段本可以提早發現這個問題？
> [回答：需求階段 / 設計階段 / Code Review / 測試階段]

### DL-03 這是需求問題、設計問題、實作問題、還是驗證問題？
> [回答：歸類 + 理由]

### DL-04 未來如何避免同類問題？
> [回答：具體措施 — 新增檢查項 / 修改模板 / 加強測試 / 調整流程]
```

### 缺陷分類標準

| 分類 | 定義 | 改善方向 |
|------|------|---------|
| 需求問題 | 需求不清、遺漏、矛盾 | 強化需求思考模板、補充訪談 |
| 設計問題 | 架構不當、介面不一致、遺漏邊界 | 強化設計思考模板、加強 G4-ENG |
| 實作問題 | 代碼邏輯錯誤、未按規格實作 | 強化提交前檢查、Code Review |
| 驗證問題 | 測試不足、遺漏情境 | 強化測試情境設計、QA 覆蓋率 |

## 與現有流程的整合

| 模板 | Pipeline | 綁定 Agent | 對應 Gate 檢查項 |
|------|----------|-----------|----------------|
| 需求思考 | P01 | PM Agent | Gate 1：G1-FT 確認每功能有需求思考紀錄 |
| 設計思考 | P02/P03 | Architect / Backend / Frontend | G4-ENG：確認每功能有設計思考紀錄 |
| 提交前檢查 | P04 | Backend / Frontend | Gate 3：抽查 commit 有對應 PC 紀錄 |
| 缺陷學習 | P04/Hotfix | QA / Review | 回顧改善：彙整所有 DL 紀錄 |

### Agent 自動觸發規則

```
PM Agent 撰寫 User Story 前：
  → 讀取 forced-thinking skill
  → 對每個功能完成「需求思考 7 問」
  → 嵌入 US 文件頂部

Architect/Backend/Frontend Agent 設計功能前：
  → 讀取 forced-thinking skill
  → 對每個功能完成「設計思考 8 問」
  → 嵌入設計文件

Backend/Frontend Agent commit 前：
  → 完成「提交前檢查 6 問」
  → 記錄至 commit message 或獨立檔案

QA Agent 記錄缺陷時：
  → 完成「缺陷學習 4 問」
  → 嵌入測試報告
```

## 強制程度與例外

| 等級 | 適用 | 規則 |
|------|------|------|
| 🔴 嚴格強制 | 需求思考、設計思考 | 不完成不得進入下一 Pipeline |
| 🟡 標準強制 | 提交前檢查 | Quick Mode 可簡化但不可省略 |
| 🟢 事後強制 | 缺陷學習 | 發現缺陷後 24hr 內必須完成 |

**例外處理**：只有 Hotfix Critical/High 可以跳過需求思考和設計思考，但必須在 48hr 補件中補齊。
