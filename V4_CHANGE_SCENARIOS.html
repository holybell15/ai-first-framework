V4.0 變更情境指南

開發過程中遇到需求變更、需求不足、新增需求時的標準處理流程

⚠️
情境 1：開發中發現需求文件寫不好
REQ_CONFIRM 某段描述模糊、矛盾或遺漏，無法直接實作
觸發時機：任何 Specialist（Backend / Frontend / DBA）在 Build 階段執行時，發現 RS 或 AC 中的需求描述與 Prototype 矛盾，或缺乏足夠資訊無法繼續實作。
1

Specialist 發出 DRIFT_SIGNAL

Specialist 發現問題後停止實作，向 Task-Master 發出 DRIFT_SIGNAL (scope_drift)，附上具體描述：哪個需求項、問題是什麼、影響範圍。

2

Task-Master 判斷退回層級

Task-Master 根據 Fix Upstream (T4) 原則判斷：問題出在需求層（REQ_CONFIRM 本身有問題）→ 退回 Discover；問題出在設計層（RS 翻譯錯誤但需求本身 OK）→ 退回 Plan。

3

凱子 確認修正方向

Task-Master 呈報問題與建議修正方式。凱子確認：是需要修改 REQ_CONFIRM（走 CIA 流程）、還是只需修正 RS/AC 即可。

4

Agent 執行修正 + 重過 Gate

退回到對應階段，修正文件後重新走 Gate Review。例如退回 Discover：更新 AC → 凱子確認 → 過 Discover Gate → 更新 RS → 過 Plan Gate → 繼續 Build。

5

Gate 記錄在 DECISIONS.md

整個修正過程記錄為一筆 Decision（例如 D45），包含：原始問題、退回原因、修正內容、凱子批准時間。確保可追溯。

關鍵原則：絕不在下游硬猜

  • 發現上游有問題 → 回上游改，改完再往下走
  • 垂直切片保護：退回只影響當前 Feature，不波及其他 Feature
  • Specialist 禁止自行「補齊」需求，只能發 DRIFT_SIGNAL
➕
情境 2：開發完成後要新增需求
Feature 已通過 Verify Gate，之後凱子想加新功能或擴充現有功能
觸發時機：凱子在使用已完成的 Feature 後，提出「我還想要 XXX 功能」或「這個 Feature 還要加一個 YYY」。
1

凱子 提出新需求

描述想要的新功能。可以是全新功能（例如「加一個報表匯出」）或擴充現有功能（例如「登入要加雙因子驗證」）。

2

Task-Master 執行 CIA（變更影響評估）

CIA 分析包含：影響哪些已完成的 Feature？需要改哪些 Baselined 文件？預估工作量多大？是否有技術風險？產出 CIA 報告。

3

凱子 審核 CIA 並決定

看完影響評估後決定：✅ 同意 → 進入實作排程；❌ 暫緩 → 放入 Backlog；🔄 調整 → 修改需求範圍後重新 CIA。

4

Task-Master 分流處理

全新功能：新增 Feature 編號（如 F14），加入垂直切片佇列尾端，從 Discover 開始跑完整週期。
擴充現有 Feature：對該 Feature 重開一輪修正週期（Discover → Plan → Build → Verify），只改受影響的部分。

5

Gate 更新 Scope Baseline

REQ_CONFIRM 更新（版本號遞增，如 v4.0.4 → v4.0.5），Prototype 如需更新也同步修改。所有變更記入 DECISIONS.md。Scope Baseline 正式擴張但有記錄。

重點：Scope Baseline「不可偷偷擴張」不代表「永遠不能改」。它的意思是——每一次擴張都必須經過 CIA → 凱子批准 → 記錄，不能由 Agent 自行決定。
🔀
情境 3：實作結果與 Prototype 不符
Build 產出的 UI 與 Baselined Prototype 長得不一樣
觸發時機：凱子在 Verify 階段對比實作結果與 Prototype，發現佈局、互動或欄位不一致。或 QA Agent 自動比對發現差異。
1

Gate Verify Gate 不通過

Verify Gate 的出口標準包含「UI 與 Baselined Prototype 一致」。不通過 = 退回 Build。

2

Frontend Agent 比對差異

逐一列出不一致的項目：缺少哪些元素、多了什麼、位置或行為不對。產出差異清單。

3

凱子 判斷

是 Bug：實作偏離了 Prototype → 修 Code，重過 Verify Gate。
Prototype 需要更新：實作其實更好 → 走 CIA 更新 Prototype Baseline，再過 Verify Gate。

4

Agent 修正後重新驗證

根據凱子的判斷執行修正（改 Code 或 CIA 更新 Prototype），然後重新進入 Verify Gate。

這就是為什麼 Prototype 要 Baseline

  • 沒有 Baseline 的 Prototype → 沒有比對基準 → 做出來長什麼樣都算「對」
  • 有 Baseline 的 Prototype → 明確的視覺合約 → 差異一目了然
  • 上一次 AICC 問題：34 項 Prototype↔RS 不同步就是缺少這道防線
🤖
情境 4：Agent 自行膨脹需求
Agent 在 RS 或 Code 中添加了凱子沒有要求的「業界最佳實踐」
觸發時機：Gate Review 時發現 RS 或 Code 包含 REQ_CONFIRM 中不存在的功能或欄位。這是上次 AICC 最大的 Root Cause（4,984 → 29,431 行膨脹）。
1

Gate Review 攔截膨脹

Plan Gate 或 Build Gate 的 Review Agent 逐條比對：RS/Code 的每個功能點是否都能 trace 回 REQ_CONFIRM 的功能編號。無法 trace 的項目標記為 ❌ Scope Creep。

2

Gate 不通過，退回修正

Gate 不通過。Agent 必須移除所有無法 trace 的內容，或者如果 Agent 認為這些內容確實必要，必須發 DRIFT_SIGNAL 讓凱子決定。

3

凱子 決定是否納入

不需要：Agent 刪除膨脹部分，重過 Gate。
確實需要：走 CIA 流程正式納入 Scope Baseline，更新 REQ_CONFIRM 版本，然後再過 Gate。

V4.0 如何預防這件事：Scope Baseline Tracing 規則要求每一行 RS/Code 都必須對應到 REQ_CONFIRM 的功能編號。沒有編號 = 不允許存在。這在舊版是完全沒有的機制。
⚡
流水線重疊（Pipeline Overlap）
在 Build 階段等待時，提前開始下一個 Feature 的 Discover
核心問題：嚴格的垂直切片（F01 全部做完才做 F02）效率太低。凱子在 F01 Build 期間無法參與，時間被浪費。
時間 →
F01: Discover Plan ═══ Build ═══ Verify
F02: · · · · · · · · · Discover Plan ═══ Build ═══ Verify
F03: · · · · · · · · · · · · · · · · · · Discover Plan ...
1

F01 進入 Build → 凱子空出來了

Agent 在寫 Code，凱子不需要參與。這段時間就是啟動 F02 Discover 的最佳時機。

2

凱子 開另一個 Session → 啟動 F02 Discover

在另一個 Claude session 中開始 F02 的需求討論、AC 確認。這是文件層工作，不碰 Code，不會跟 F01 Build 衝突。

3

F02 可以推進到 Plan（但不能進 Build）

F02 的 RS 和 Tech Spec 都可以寫。但 F02 不能進入 Build，直到 F01 通過 Verify Gate。原因：兩個 Feature 同時改 Code 會互相衝突。

4

F01 Verify 通過 → F02 進入 Build

F01 驗收完成。F02 已經走完 Discover + Plan，可以立刻進入 Build，不用等。效率提升 30-50%。

硬性限制

  • 同時只能有 1 個 Feature 在 Build（Code 層互斥）
  • Discover 和 Plan 可以跟任何階段並行（文件層不衝突）
  • 每個 Feature 的 Gate 獨立驗證，不因並行而降低標準
  • 如果 F01 Build 發生退回（DRIFT_SIGNAL），F02 的 Discover/Plan 不受影響
實際操作方式：F01 在 Session A 跑 Build。凱子開 Session B，對同一個 AICC-III 專案開始 F02 的 Discover。兩個 Session 各自讀寫不同的文件（F01 改 Code，F02 寫 AC/RS/Tech Spec），不會衝突。
📐
技術規格書（Tech Spec）
填補「需求」到「Code」之間的技術設計空白
解決的問題：目前文件鏈是 REQ → AC → RS → Code。RS 寫「做什麼」但沒寫「怎麼做」。導致不同 Feature 的 Agent 對 API 設計、Schema、共用元件各做各的，久了就不一致。

文件鏈（更新後）

REQ_CONFIRM → AC 驗收標準 → RS 需求規格 → Tech Spec 🆕 → Code

Tech Spec 涵蓋內容

API 設計
端點清單、Request/Response 規格、認證授權方式、Error Code 定義
資料模型
Schema 設計、欄位型別與約束、索引策略、Migration 計畫
前端架構
元件結構、狀態管理、路由規劃、Prototype 對應表
共用元件
本 Feature 提供的共用元件 + 依賴的其他 Feature 元件（最關鍵！）
技術決策
選型理由、與既有 DECISIONS.md 的一致性驗證
安全與測試
安全性考量、輸入驗證、測試策略與覆蓋率目標

Tech Spec 在流程中的位置

1

Plan 階段：RS 確認後

凱子確認 RS（做什麼）後，Architect Agent 根據 RS 撰寫 Tech Spec（怎麼做）。命名：TS_F[XX]_[名稱]_v0.1.md

2

Architect 讀取已有 Tech Spec

如果不是第一個 Feature，必須讀取所有已 Baselined 的 Tech Spec，確保 API 風格、Schema 命名、共用元件使用方式保持一致。

3

凱子 確認 Tech Spec → Baseline

凱子確認後，Tech Spec 狀態變為 Baselined。從此修改必須走 CIA。

4

Plan Gate 新增檢查項

Plan Gate 通過條件新增：Tech Spec 已完成且凱子確認 + 與既有 Tech Spec 不衝突 + 每個技術項目都追溯到 REQ。

5

Build 階段：Agent 必須遵循 Tech Spec

Backend / Frontend / DBA Agent 在 Build 時以 Tech Spec 為技術依據。不能偏離，偏離就是 DRIFT_SIGNAL。

Tech Spec 解決的根本問題

  • 跨 Feature 一致性：F05 的 Agent 知道 F01 用了 JWT，不會自己搞一套 Session
  • 共用元件管理：Auth Service 是 F01 建的，F02 要用就參照 F01 的 Tech Spec §5
  • 技術決策可追溯：為什麼選 Pinia 不選 Vuex？Tech Spec §6 有記錄
  • 修改有保護：改 F01 的 API 格式 → CIA → 評估影響 F02、F03 → 凱子批准
模板位置：02_Specifications/TEMPLATE_Tech_Spec.md — 已放入 V4.0 Template 和 AICC-III 專案。

舊做法 vs V4.0 做法

同樣一個問題，在舊版和 V4.0 分別會怎麼處理

✗ 舊做法（AICC NEW 360）

  • ❌ 需求模糊 → Agent 自己猜測補齊 → 膨脹
  • ❌ 33 份 RS 同時展開 → 改一個影響全部
  • ❌ 沒有 Scope Baseline → 不知道什麼算「範圍內」
  • ❌ Prototype 僅供參考 → 做出來長不一樣也沒人發現
  • ❌ Gate 存在但不嚴格 → 膨脹的 RS 直接通過
  • ❌ 一天做 9 個決策 → 沒有影響評估
  • ❌ 問題在 audit 才發現 → 修正成本 10 倍
  • ❌ 4,984 → 29,431 行 → 誰加的？為什麼加？不知道
  • ❌ 沒有技術規格書 → 不同 Agent 各做各的技術決策
  • ❌ 嚴格串行 → 凱子在 Build 期間只能等

✓ V4.0 做法（AICC-III）

  • ✅ 需求模糊 → DRIFT_SIGNAL → 退回上游釐清
  • ✅ 垂直切片 → 一次一個 Feature → 影響範圍最小
  • ✅ Scope Baseline 明確 → 凡事 trace 回 REQ_CONFIRM
  • ✅ Prototype = Baseline → UI 必須與 Prototype 一致
  • ✅ Gate 強制 Tracing 驗證 → 膨脹在 Gate 被攔截
  • ✅ 每次變更先做 CIA → 凱子批准才能改
  • ✅ Fix Upstream → 問題在哪層就在哪層修
  • ✅ 每次 Scope 變更都記入 DECISIONS.md → 完全可追溯
  • ✅ Tech Spec 統一技術設計 → 跨 Feature 一致性有保障
  • ✅ 流水線重疊 → Build 期間可提前 Discover 下一個 F
