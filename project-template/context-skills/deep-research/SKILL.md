# Deep Research Skill

## 研究流程（4 步）

### Step 1 — 定義問題範圍

在開始搜尋前，先清楚自己問的是什麼：

```markdown
**要比較的東西**：[技術名稱] vs [技術名稱] vs ...

**為什麼要做這個決策**：[背景 / 痛點]
- 例：「目前用 REST API，但實時性要求高，考慮 WebSocket」

**評估標準**（權重）：
- 效能（30%）
- 開發速度（25%）
- 社群成熟度（20%）
- 安全性（15%）
- 學習曲線（10%）

**預期產出**：ADR / 比較表 / 深度研究報告

**決策截止**：[日期，例如周五下班前]
```

### Step 2 — 資料蒐集

不是全網搜尋，而是有優先順序的蒐集：

1. **官方文件** — 架構決策、最佳實踐、性能基準（最可信）
2. **社群經驗** — GitHub issue、Stack Overflow、掘金文章（要看時間戳，3 年前的資料可能過時）
3. **Benchmark** — 性能測試對比、壓測報告（同樣規格下的數據）
4. **案例研究** — 大公司怎麼用（可信度高但可能是特殊場景）

**紀錄方式** — 每一筆來源都要註記：

```
[URL]
→ 發現：[該來源說了什麼]
→ 可信度：⭐⭐⭐（高/中/低 + 原因，例如「Google 官方文件」或「個人部落格，可能過時」）
```

### Step 3 — 比較矩陣

把所有發現整理成表格，對標統一的評估標準：

```markdown
| 標準 | Vue 3 | React 18 | Angular 17 | 權重 |
|------|--------|----------|-----------|------|
| **效能** | 頂級編譯優化 | FRP 機制有開銷 | 完整框架 | 30% |
| **開發速度** | 語法簡潔，入門快 | 需懂 JSX + Hook 生態 | 陡峭學習曲線 | 25% |
| **社群** | 國內活躍 | 全球最大 | 企業級但小眾 | 20% |
| **安全** | 無特殊風險 | React 本身無，生態參差不齊 | 默認較安全 | 15% |
| **學習** | 2-3 周上手 | 1 周基礎，3 月精通 | 2-3 月 | 10% |
| **小計** | | | | 100% |
```

**打分邏輯**：
- 定性評分（好/中/差）或 1~5 分，統一用數字
- 乘以權重後加總，得到加權分數
- 分數僅供參考，決策還要考量「當下團隊能力」和「專案時限」

### Step 4 — ADR 產出並存檔

決定了後，寫入 ADR（Architecture Decision Record）並存到 `memory/decisions.md`：

```markdown
## ADR-[編號]: [決策標題]

**狀態**：Proposed / Accepted / Deprecated

**背景**
為什麼要做這個決策？遇到什麼問題或機遇？

**決策**
選了什麼？（一句話）

**原因**
為什麼選它？哪些因素最關鍵？
- 效能贏 X%
- 社群規模 5 倍大
- 團隊已有經驗
- 合規需求

**後果**
這個決策的影響是什麼？
- 積極：[能帶來什麼好處]
- 消極：[增加什麼風險或成本]

**替代方案及否決原因**
| 方案 | 為什麼沒選 |
|------|----------|
| [方案A] | [原因] |
| [方案B] | [原因] |

**相關決策**（若有）
- 相關的上下游決策
- 例：ADR-003（DB 選型）與本決策耦合

**參考資料**
- [來源1] — 說了什麼
- [來源2] — 說了什麼
```

**例子**：

```markdown
## ADR-001: 選用 Vue 3 作前端框架

**狀態**：Accepted

**背景**
專案要在 3 個月內完成原型。團隊有 2 位 Vue 經驗者，0 位 React 熟手。

**決策**
採用 Vue 3 + Vite。

**原因**
- 開發速度：Vue 語法簡潔，團隊已有經驗，無學習成本
- 編譯速度：Vite 冷啟動 < 1s，遠快 webpack
- 社群：國內活躍，遇到問題好找資源

**後果**
- 積極：預計開發週期短 1 周
- 消極：海外招募工程師時，React 人才池更大

**替代方案**
| 方案 | 否決原因 |
|------|--------|
| React 18 | 生態太多選擇（CSS-in-JS 方案 20+），3 月內無法穩定 |
| Angular 17 | 陡峭學習曲線，團隊時間不允許 |

**參考資料**
- Vue 3 官方基準測試（2024） — 編譯速度最佳
- Vite 性能報告 — 開發環境啟動 < 1s
- GitHub Trending Vue/React 對比 — Vue 國內使用率更高
```

## GSD 鉤點

- **Architect 用**：技術選型時觸發，寫入 ADR 到 `memory/decisions.md`
- **Security 用**：合規調研後寫「合規決策 ADR」
- **Gate 2 Review**：檢查所有架構決策是否都有 ADR + 替代方案
- **新成員 onboarding**：讀 `memory/decisions.md` 理解為什麼做這些決策

## 常見誤區

| 錯誤 | 正確做法 |
|-----|--------|
| Google 第一筆搜尋結果就當真 | 多找 3-5 個來源交叉驗證 |
| 數字越高越好 | 要看權重，有時「學習快」比「效能高」更重要 |
| 決定了就鎖死 | ADR 寫「可重新評估條件」，例如「若社群衰退 → 可改用 X」 |
| 忽視團隊能力 | 最好的技術也要團隊會用，否則當知識產生 |

## 快速參考

```bash
# 開始調研
1. 寫 Research Brief（Step 1）
2. 蒐集資料（Step 2）
3. 填比較矩陣（Step 3）
4. 寫 ADR（Step 4）

# 寫入決策檔
echo "## ADR-[N]: [標題]" >> memory/decisions.md
[複製 ADR 內容]

# 驗證
grep "ADR-" memory/decisions.md | wc -l
# 應該不少於 3 條 ADR（重要決策都要記）
```
